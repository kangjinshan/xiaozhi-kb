# Wi-Fi 网络麦克风方案

本文记录「把 ESP32-C6 板子做成 MacBook 的无线麦克风」的技术方案、原理和落地思路。

## 背景与结论

需求:希望这块 Waveshare ESP32-C6-Touch-AMOLED-2.16 在当蓝牙 HID 键盘的同时,也能给 MacBook 当麦克风。

先说被否掉的路线:**蓝牙麦克风做不到**。原因:

- 传统蓝牙麦克风走 Classic Bluetooth 的 HFP/HSP 协议,而 ESP32-C6 只支持 BLE,没有 Classic Bluetooth。
- BLE 上的替代标准 LE Audio(BLE + LC3 + ISO 通道)在 ESP-IDF 端的麦克风/音频源方向不成熟,macOS 端也难把第三方 BLE 设备识别成系统麦克风输入。

可行的无线路线是 **Wi-Fi 网络麦克风**:ESP32 采集音频,通过 Wi-Fi 用 UDP 把音频流推给 MacBook,MacBook 端把它变成一个系统麦克风输入设备。

## 核心原理:同一内网 + 端口发送

关键认知:**Wi-Fi 连路由器 ≠ 和某个设备一对一绑定,而是「接入一个局域网」。**

- 路由器相当于一栋楼的前台。ESP32 和 MacBook「连上路由器」= 各自在前台登记了一个房间号(内网 IP)。
- 登记完之后,同一内网内的设备可以**直接互发数据**,不需要再「连接」对方,也不需要 MacBook 连多个 Wi-Fi。
- MacBook 只连一个 Wi-Fi(和平时上网同一个)即可。

```
        路由器 (192.168.1.1)
        /              \
   ESP32-C6          MacBook
 192.168.1.50      192.168.1.66
        \______直接互发______/
```

通信三要素:

- **IP**:定位哪台设备(同内网用内网 IP,无需公网或内网穿透)。
- **端口**:定位设备上的哪个程序(双方约定一个,例如 6000)。
- **协议**:实时音频用 **UDP**(低延迟,丢包不重发);需要可靠传输才用 TCP。

数据流:

```
同一内网 (192.168.1.x)
ESP32 ──把音频UDP包发到──▶ 192.168.1.66 : 6000
                                    ▲
                          MacBook 的接收程序监听这个端口
```

## 组网方式

### 方式 A:两者都连同一路由器(推荐)

ESP32 和 MacBook 都连家里同一个 Wi-Fi,拿到同一网段 IP,直接互通。MacBook 仍可正常上外网。

### 方式 B:ESP32 自己开热点,MacBook 连它

ESP32 开 AP 热点,MacBook 连这个热点。缺点:此时 MacBook 上不了外网(macOS 同一时刻只用一个 Wi-Fi)。仅在无路由器环境下考虑。

## 端到端链路

```
ESP32 麦克风 → I2S采集 → UDP over Wi-Fi
                                  ↓
MacBook: 接收程序 → 写入 BlackHole 虚拟声卡 → 系统麦克风输入
```

### ESP32-C6 端(固件)

1. **采集音频**:读板载麦克风的 I2S PCM 数据(小智语音模式已在用,可复用)。
2. **编码/打包**:原始 PCM 或压缩后,按 UDP 包组织。
3. **发送**:通过 Wi-Fi 把音频流 UDP 推到 MacBook 的 IP:端口。

小智语音模式已有「麦克风采集 + Wi-Fi 联网」的现成代码,可复用大半。

### MacBook 端(关键难点)

macOS 不会自动把网络音频流当成麦克风,需要两个东西配合:

1. **接收程序**:在 Mac 上跑一个小程序,监听 UDP 端口,收流并解码成 PCM(需自己写)。
2. **虚拟音频设备**:装 [BlackHole](https://github.com/ExistentialAudio/BlackHole)(免费开源虚拟声卡)。接收程序把音频写进 BlackHole,它就会在「声音输入」里显示成一个麦克风。
3. 在 macOS「系统设置 → 声音 → 输入」里选这个虚拟设备,Zoom/录音软件即可使用。

## ESP32 如何找到 MacBook 的 IP

唯一需要额外处理的问题,常见三种做法:

| 做法 | 原理 | 优缺点 |
| --- | --- | --- |
| 写死 IP | 固件里填死 MacBook 的内网 IP | 最简单;MacBook IP 变了要改 |
| MacBook 主动连 ESP32 | ESP32 当服务器,Mac 端程序去连它 | ESP32 IP 也会变,可配合 mDNS |
| mDNS 自动发现 | ESP32 广播名字(如 `xiaozhi-kb.local`),Mac 按名字找 | 最省心,IP 变了也能自动找到;开发量略大 |

## 现实评估

| 维度 | 情况 |
| --- | --- |
| 无线 | ✅ 真无线(Wi-Fi) |
| Mac 需装东西 | ⚠️ 要装 BlackHole + 跑一个接收程序(需自己写) |
| 延迟 | ⚠️ Wi-Fi + 缓冲,通常几十~上百 ms,打字/开会够用,专业录音偏高 |
| 开发量 | 🔴 中等偏大:ESP32 端推流 + Mac 端接收程序两头都要写 |
| 和键盘并存 | ✅ Wi-Fi 与 BLE HID 键盘可同时跑(C6 支持 Wi-Fi+BLE 共存,但资源紧张,需实测) |

## 与 USB 复合设备方案对比

如果能接受插线,**USB 复合设备(HID 键盘 + UAC 麦克风)** 比 Wi-Fi 方案简单得多、延迟更低:

| 方案 | 连接方式 | 可行性 | 说明 |
| --- | --- | --- | --- |
| USB 复合设备(键盘+麦克风) | USB 线 | ✅ 最现实 | ESP32-C6 自带 USB,可做 HID + USB Audio Class 复合设备,插线即用;需解决 USB 口与烧录/串口共存 |
| Wi-Fi 网络麦克风 | Wi-Fi | ⚠️ 需 Mac 端软件 | 本文方案,真无线但两头都要开发 |
| BLE 蓝牙麦克风 | 蓝牙 | ❌ | C6 无 Classic Bluetooth,LE Audio 不成熟 |

## 待确认的决策点

1. 麦克风和蓝牙键盘是否需要同时使用?若是,ESP32 需同时开 Wi-Fi(推音频)+ BLE(HID 键盘),C6 能共存但资源紧张,需实测。
2. 能否接受在 Mac 上装 BlackHole + 跑一个后台接收程序?(macOS 无零依赖方案)
3. 若只是偶尔用、不介意插线,建议重新对比 USB 复合设备方案。
